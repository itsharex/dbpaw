# Issue #84 需求分析：侧边栏展示函数、存储过程等数据库对象

> 背景：有用户反馈连接 SQL Server 后，侧边栏看不到函数、存储过程等对象。本文从需求层面评估是否有必要支持，以及支持的策略。

---

## 一、现状

当前 DbPaw 的侧边栏统一只展示以下层级：

```
Connection → Database → [Schema] → Table/View → Column
```

**所有 8 种数据库均不支持函数、存储过程、触发器等对象的浏览。**

- 后端 `DatabaseDriver` trait 仅有 `list_tables` 方法
- 前端树形结构无 Routine/Function/Procedure 节点类型
- API 层无 `listRoutines` / `listProcedures` 等接口

---

## 二、需求拆解：用户到底想要什么？

issue 描述为"看不到函数、存储过程等"，实际诉求可分为四个层级：

| 层级 | 需求 | 用户价值 | 实现复杂度 |
|------|------|---------|-----------|
| **L1：浏览** | 侧边栏能展开看到有哪些 Function / Procedure | 知道数据库里有什么 | 低 |
| **L2：查看定义** | 点击后能看到 `CREATE PROCEDURE` / `CREATE FUNCTION` 的完整 DDL | 理解业务逻辑 | 中 |
| **L3：执行** | 能运行 `EXEC sp_xxx` 并查看结果 | 调试/验证 | 中（复用现有执行能力） |
| **L4：编辑** | 能修改定义、保存、调试 | 开发维护 | **高** |

**判断**：从 issue 的简短描述来看，用户最可能的诉求是 **L1 + L2**（浏览 + 查看定义），与当前"点表看列"的复杂度处于同一级别。

---

## 三、不同数据库的诉求差异

| 数据库 | 存储过程/函数使用率 | 在开发流程中的地位 |
|--------|---------------------|------------------|
| **SQL Server** | **非常高** | 企业级应用大量用 SP 封装业务逻辑，DBA 日常必看 |
| **PostgreSQL** | 高 | 函数是核心特性，很多扩展依赖它 |
| **MySQL 家族** | 中低 | 大部分 Web 开发直接用 ORM，手写 SP 的相对较少 |
| **SQLite** | 极低 | 几乎不用 |
| **Redis** | 不适用 | — |

**关键结论**：
- 对 **SQL Server 用户**而言，看不到函数/存储过程是**基础功能缺失**，不是边缘需求。
- 对 MySQL 用户而言，这是"锦上添花"。
- 如果 DbPaw 想成为一个"称职的 SQL Server 客户端"，支持该功能是必要的。

---

## 四、竞品对标

| 工具 | 是否展示 SP/Function | 体验 |
|------|---------------------|------|
| **DBeaver** | ✅ 有专门节点 | 完整，可浏览/查看/编辑/调试 |
| **Navicat** | ✅ 有专门节点 | 完整 |
| **DataGrip** | ✅ 有专门节点 | 完整 |
| **Azure Data Studio** | ✅ 核心功能 | 微软官方工具，SP 支持是核心卖点 |
| **TablePlus** | ✅ 有 Routines 节点 | 轻量，仅列出 + 查看 DDL |
| **HeidiSQL** | ✅ 有 Routines 标签 | 轻量 |

**结论**：即使是 TablePlus、HeidiSQL 这类以简洁著称的工具，也支持了函数/存储过程的浏览。DbPaw 当前不支持，在 SQL Server 场景下是明显短板。

---

## 五、产品定位评估

DbPaw 的定位是**"轻量但够用"**。

- 如果只做 **L1（浏览列表）+ L2（查看 DDL）**，与当前"点表看列"的复杂度完全一致，**不违背轻量定位**。
- L3/L4（编辑、调试）确实会变重，但那是后续可选项，当前无需承诺。

"列出数据库对象并显示其定义"是数据库客户端的基础能力，不应被视为"重型功能"。

---

## 六、布局策略评估：只放在 tree 布局？

用户提出"只保留在传统的 tree 布局上，默认 tabs 布局保持简洁"。

### 6.1 tree 布局的当前结构

```
Connection
├── Queries
└── Databases
    └── Database
        ├── Schema
        │   └── Table → Column
        └── Table → Column
```

### 6.2 两种策略对比

| 策略 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A：tree 布局专属** | 仅在 `layoutMode="tree"` 时展示 Routines 节点 | 默认用户不受影响；可作为"高级视图"实验 | 默认布局用户无法使用；产品能力不一致；issue #84 的用户大概率在用默认布局 |
| **B：统一支持** | tabs 和 tree 布局均支持，仅在 tree 下组织更好看 | 所有用户受益；产品一致性高；issue 得到真正解决 | 默认布局也引入了新节点，"简洁度"略有下降 |

### 6.3 关键风险

- **tree 布局是非默认的、可选的**。大部分用户可能根本不知道有这个模式。
- 如果功能只藏在 tree 布局里，对默认布局用户等于**不存在**。
- 同一连接在不同布局下能看到的对象类型不同，会造成困惑。

**结论**：**不建议将 Routines 浏览做成 tree 布局专属**。tree 布局适合"组织架构差异"（如把 Queries 和 Databases 放一棵树），不适合"功能有无差异"。

## 七、实现成本速览（若决定做）

若仅支持 **MSSQL + L1/L2（浏览 + 查看 DDL）**，预估工作量：

| 模块 | 工作 | 量级 |
|------|------|------|
| 后端数据模型 | 新增 `RoutineInfo` 类型 | 小 |
| 后端驱动 trait | `DatabaseDriver` 扩展 `list_routines`（可选实现） | 小 |
| 后端 MSSQL 实现 | 查 `sys.objects` + `sys.sql_modules` | 中 |
| 前端 API | `api.ts` 新增 `listRoutines` | 小 |
| 前端数据结构 | `SchemaInfo` / `DatabaseInfo` 扩展 | 小 |
| 前端状态管理 | 新增展开态、懒加载逻辑 | 中 |
| 前端渲染 | 侧边栏新增 Routines 节点（支持 Schema / 不支持 Schema 两种层级） | 中 |
| 前端 Adapter | `DatasourceTreeAdapter` 扩展驱动能力声明 | 小 |
| DDL 查看 | 双击/点击节点查看 `CREATE PROCEDURE` 定义 | 中 |

**总计：约 2-3 天（单 MSSQL，仅浏览 + DDL 查看）**

后续给 PostgreSQL / MySQL 补上的成本很低（都查 `information_schema.routines`，SQL 类似）。

---

## 八、综合结论与建议

### 8.1 是否有必要添加？

**有必要，且建议作为通用能力（不限制在 tree 布局）。**

理由：
1. **需求真实**：SQL Server 生态中 SP/Function 是基础设施，不是边缘需求。
2. **竞品标配**：连轻量级竞品（TablePlus / HeidiSQL）都支持了。
3. **成本可控**：仅做到"浏览 + 查看 DDL"，与现有"看表结构"同复杂度。
4. **定位不冲突**：这是基础能力补齐，不是重型功能堆砌。
5. **tree 专属无意义**：默认布局用户占大多数，藏到 tree 里 issue 依然得不到解决。

### 8.2 建议的实施策略

**若决定做：**
1. **第一阶段**：支持 MSSQL 的 Functions & Procedures 浏览 + DDL 查看。
2. **第二阶段**：低成本扩展到 PostgreSQL / MySQL（复用 `information_schema.routines`）。
3. **暂不做**：编辑存储过程、调试、触发器/序列等更复杂对象。

**若决定不做：**
- 在 issue #84 下明确回复："当前所有数据库均仅支持表/视图浏览，函数/存储过程的支持已纳入 Roadmap，后续统一提供。"
- 避免只给 tree 布局做 patch，以免造成产品能力割裂。
