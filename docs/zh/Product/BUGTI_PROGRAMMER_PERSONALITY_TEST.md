# BUGTI 程序员 Bug 型人格测试方案

## 1. 产品定位

BUGTI 是一个面向程序员、技术从业者和技术社区用户的娱乐向人格测试。

它不是测能力，也不是做职业评估，而是用程序员黑话、代码梗、线上事故梗和职场协作梗，生成一种“有点冒犯但又有点准”的代码精神状态结果。

建议产品名：

- BUGTI：Bug Type Indicator
- SBugTI：程序员发疯人格测试
- 码病测试
- 程序员精神状态诊断

推荐主名称：

> BUGTI：程序员 Bug 型人格测试

推荐副标题：

> 30 道题，测出你的代码精神状态。
>
> 仅供娱乐，请勿用于绩效、招聘、相亲、背锅分配。

核心目标：

- 让程序员愿意主动完成测试。
- 让结果页适合截图和转发。
- 让用户看到结果时产生“虽然在骂我，但确实像我”的感觉。
- 后续可以叠加 GitHub 数据，做“自我认知 vs 真实代码记录”的反差复判。

## 2. 核心体验

用户流程：

1. 进入测试页。
2. 输入昵称，可跳过。
3. 回答 30 道程序员情境题。
4. 中途可能触发隐藏人格。
5. 得到人格结果。
6. 生成分享图。
7. 可选输入 GitHub 用户名，解锁“现实复判”。

第一版建议做纯前端答题，不需要登录，不需要后端，不收集用户数据。

## 3. 设计原则

### 3.1 结果标签要强

不要使用太正常、太职业化的标签，例如“架构师”“交付型工程师”“协作型开发者”。

应该使用更像互联网梗图的标签，例如：

- YAML 缩进奴
- FORCE 推土机
- TODO 欠债者
- PROD 生产祭品
- ENV 环境人
- TEST 测试鬼
- README 体面人

### 3.2 题目要来自真实程序员场景

题目应该围绕：

- 线上报警
- PR review
- CI/CD
- 环境变量
- 联调
- 技术债
- 需求变更
- 重构
- 测试
- 日志
- 缓存
- Docker
- Git 冲突
- 估时

### 3.3 不直接问人格

不要问：

> 你是一个谨慎的人吗？

应该问：

> 你看到 `--force` 会怎么做？

### 3.4 适度冒犯，但不做人身攻击

可以用自嘲式、技术梗式、精神状态式文案。

避免真实歧视、地域、性别、年龄、学历、职业能力贬损。

## 4. 内部测试模型

用户侧不需要看到复杂模型，内部使用 6 个维度计算。

### 4.1 交付执念

衡量用户更像“先上线再说”，还是“这个坑我先挖着”。

高分倾向：

- 业务优先
- 能跑就行
- 线上先救
- 喜欢 MVP
- 不怕临时方案

低分倾向：

- 容易拖延
- 喜欢规划
- 项目永远 WIP
- TODO 越写越多
- 总想等完美再发布

相关人格：

- 高：MVP、HOTFIX、PROD、CRUD
- 低：TODO、WIP、YAK、README

### 4.2 规范洁癖

衡量用户对测试、类型、格式化、文档和工程规范的执念。

高分倾向：

- 不能忍 `any`
- 不能忍无测试
- 不能忍格式混乱
- 会为命名纠结很久
- CI 红了会睡不着

低分倾向：

- 先跑起来
- 手动测一下就行
- 类型以后再补
- README 随缘
- 格式化不影响业务

相关人格：

- 高：TEST、YAML、README、JSON
- 低：ANY、MVP、FORCE、BUG

### 4.3 风险偏好

衡量用户面对未知问题时，是谨慎排查，还是直接莽。

高分倾向：

- 敢 `sudo`
- 敢改生产
- 敢 `--force`
- 敢线上热修
- 敢删库前说“应该没事”

低分倾向：

- 先备份
- 先看日志
- 先灰度
- 先写测试
- 先问清楚影响范围

相关人格：

- 高：ROOT、SUDO、FORCE、HOTFIX
- 低：LOG、TEST、CACHE、CRON

### 4.4 抽象冲动

衡量用户是否喜欢架构、封装、重构、造轮子。

高分倾向：

- 一个按钮也要设计状态机
- 修 bug 顺手重构三层
- 喜欢框架、范式、模式
- 项目未上线，目录结构已进化
- 能把简单需求讲成系统工程

低分倾向：

- CRUD 就 CRUD
- 少搞抽象
- 能复制就不封装
- 先把页面做出来
- 不为未来过度设计

相关人格：

- 高：YAK、JSON、LEGACY、REGEX
- 低：CRUD、MVP、ANY、HOTFIX

### 4.5 协作状态

衡量用户是团队协作型，还是独立闭关型。

高分倾向：

- 喜欢讨论方案
- 会认真 review
- 会写文档
- 会解释上下文
- 愿意对齐需求

低分倾向：

- 不爱开会
- 不爱解释
- PR 一扔就消失
- 喜欢自己默默改
- 出问题才出现

相关人格：

- 高：LGTM、MERGE、PMD、README
- 低：404、NULL、SOLO、WIP

### 4.6 精神稳定度

衡量用户面对 bug、需求变化、线上事故时的心理状态。

高分倾向：

- 有排查节奏
- 能接受混乱
- 问题来了先分层
- 不轻易破防
- 有长期维护意识

低分倾向：

- 容易爆炸
- 容易自我怀疑
- 容易进入死循环
- 容易深夜重构
- 看到报错直接沉默

相关人格：

- 高：LOG、TEST、CRON、README
- 低：500、NaN、LOOP、STACK

## 5. 32 种核心人格

### 5.1 BUG 虫人

一句话：不是你写 bug，是 bug 借你手降临人间。

典型症状：

- 修一个 bug，长出三个新 bug。
- 你越自信，线上越危险。
- 你说“小改一下”，团队开始沉默。
- 测试环境没问题，生产环境会自己表演。

经典台词：

- “奇怪，我没动这里啊。”
- “这不应该啊。”
- “我本地是好的。”

适合搭档：TEST 测试鬼、LOG 日志仙

天敌人格：PROD 生产祭品、MERGE 冲突体

### 5.2 TODO 欠债者

一句话：TODO 写得越多，人生越有盼头。

典型症状：

- 项目里全是 `TODO: later`。
- later 从来没来过。
- 每次重构都像还高利贷。
- 你不是没做，你是在规划未来。

经典台词：

- “这个后面优化。”
- “先记个 TODO。”
- “技术债以后统一还。”

适合搭档：CRON 定时人、TEST 测试鬼

天敌人格：HOTFIX 临终抢救者

### 5.3 ANY 任意人

一句话：类型是什么？能跑就是类型。

典型症状：

- TypeScript 写得像 JavaScript。
- `any` 是你的止痛药。
- 泛型让你心烦。
- 编译器劝不住你。

经典台词：

- “先 any 一下。”
- “类型后面补。”
- “别让类型影响开发效率。”

适合搭档：MVP 糊弄者、CRUD 增删改查牲

天敌人格：TEST 测试鬼、YAML 缩进奴

### 5.4 NULL 空指针

一句话：你不是没有想法，你只是还没初始化。

典型症状：

- 启动很慢。
- 被问进度时大脑返回 null。
- 常常沉默，但不是没听见。
- 需要一个明确入口函数。

经典台词：

- “我想一下。”
- “等我理一下。”
- “这个信息还不够。”

适合搭档：README 体面人、LOG 日志仙

天敌人格：PMD 产品受害者

### 5.5 NaN 非数人

一句话：你看起来正常，参与计算后全乱了。

典型症状：

- 需求理解有自己的分支。
- 估时永远不能参与数学运算。
- debug 到最后开始怀疑人生。
- 状态不可预测。

经典台词：

- “这怎么算出来的？”
- “我刚才不是这个意思。”
- “这东西有点玄学。”

适合搭档：LOG 日志仙

天敌人格：CRON 定时人

### 5.6 404 失踪者

一句话：需求找不到你，文档也找不到你。

典型症状：

- 消息已读不回。
- PR 提完就消失。
- issue 里 @ 不出来。
- 关键时刻进入离线模式。

经典台词：

- “刚看到。”
- “我以为你们在说别人。”
- “这个我晚点看。”

适合搭档：CRON 定时人

天敌人格：PMD 产品受害者、BOSS 技术负责人

### 5.7 500 内爆者

一句话：表面服务正常，内部已经炸穿。

典型症状：

- 外表冷静，内心 HTTP 500。
- 需求一变就开始冒烟。
- 表情稳定，心态宕机。
- debug 时会突然沉默。

经典台词：

- “没事。”
- “问题不大。”
- “我处理一下。”

适合搭档：LOG 日志仙、README 体面人

天敌人格：HOTFIX 临终抢救者

### 5.8 ROOT 根人

一句话：权限给你，你能把系统和人生一起删了。

典型症状：

- 遇事先提权。
- 对危险命令有奇怪亲切感。
- 权限不够会让你暴躁。
- 你相信自己，也相信备份不存在。

经典台词：

- “给我 root。”
- “我直接上机器看看。”
- “应该不会有事。”

适合搭档：LOG 日志仙

天敌人格：TEST 测试鬼、CRON 定时人

### 5.9 SUDO 强行者

一句话：不知道为什么不行，但加权限就好了。

典型症状：

- 解决问题靠权限压制。
- 看到 permission denied 会兴奋。
- 你不排查，你突破。
- 喜欢用最短路径绕过规则。

经典台词：

- “sudo 试试。”
- “权限问题。”
- “先跑起来再说。”

适合搭档：MVP 糊弄者

天敌人格：YAML 缩进奴

### 5.10 LGTM 看过者

一句话：你 review 了，但又好像没 review。

典型症状：

- 三秒看完 800 行 diff。
- 最常评论：LGTM。
- 出问题时说“我只是看了下”。
- 你相信同事，也相信命运。

经典台词：

- “LGTM。”
- “整体没问题。”
- “细节你再看看。”

适合搭档：MERGE 冲突体

天敌人格：BUG 虫人

### 5.11 WIP 半成者

一句话：你的项目永远在路上，从不到站。

典型症状：

- 分支名永远叫 `wip`。
- demo 很多，发布很少。
- 灵感启动，收尾失踪。
- README 第一行写着 work in progress。

经典台词：

- “快好了。”
- “还差最后一点。”
- “我再 polish 一下。”

适合搭档：MVP 糊弄者、CRON 定时人

天敌人格：PROD 生产祭品

### 5.12 MVP 糊弄者

一句话：先上线，后面再说，后面从没来过。

典型症状：

- 能跑就发。
- 临时方案长命百岁。
- demo 最终变生产。
- 你把以后优化当架构原则。

经典台词：

- “先这样。”
- “MVP 嘛。”
- “用户先用起来。”

适合搭档：HOTFIX 临终抢救者

天敌人格：TEST 测试鬼、YAML 缩进奴

### 5.13 YAK 剃牦牛者

一句话：修一个按钮，顺手重写构建系统。

典型症状：

- 问题没解决，基础设施升级了。
- 小需求引发大重构。
- 总能找到更底层的问题。
- 今天只是想改颜色，明天已经在写框架。

经典台词：

- “根因是架构不合理。”
- “我顺便重构一下。”
- “这个要从底层解决。”

适合搭档：README 体面人

天敌人格：PMD 产品受害者、MVP 糊弄者

### 5.14 CRON 定时人

一句话：每天固定崩溃，像任务调度一样准。

典型症状：

- 工作节奏稳定。
- 崩溃也很规律。
- 喜欢自动化。
- 讨厌不可控变量。

经典台词：

- “这个可以自动化。”
- “每天跑一次就行。”
- “定时任务挂了？”

适合搭档：LOG 日志仙、TEST 测试鬼

天敌人格：NaN 非数人

### 5.15 YAML 缩进奴

一句话：两个空格，毁你一生。

典型症状：

- 对缩进极其敏感。
- CI 配置写到怀疑人生。
- 你的人生被冒号和空格统治。
- 看见 tab 会呼吸困难。

经典台词：

- “这里多了一个空格。”
- “YAML 又炸了。”
- “谁改了 CI？”

适合搭档：TEST 测试鬼

天敌人格：ANY 任意人、SUDO 强行者

### 5.16 JSON 括号人

一句话：外表结构化，内心全是嵌套。

典型症状：

- 喜欢层级和配置。
- 什么都想 schema 化。
- 表面清晰，实际复杂。
- 最怕少一个逗号。

经典台词：

- “这个结构不太对。”
- “配置抽出来吧。”
- “这里可以嵌一层。”

适合搭档：YAML 缩进奴

天敌人格：MVP 糊弄者

### 5.17 REGEX 咒术师

一句话：你写的不是表达式，是封印术。

典型症状：

- 一行代码没人敢改。
- 你自己三天后也看不懂。
- 用符号召唤结果。
- debug 像驱邪。

经典台词：

- “我写个正则。”
- “应该能匹配。”
- “别动这行。”

适合搭档：TEST 测试鬼

天敌人格：README 体面人

### 5.18 CACHE 缓存怪

一句话：不是缓存问题？那一定是更深的缓存问题。

典型症状：

- 万物皆可缓存。
- 万错皆疑缓存。
- 刷新、清理、重启是三件套。
- 你不相信实时数据。

经典台词：

- “清缓存试试。”
- “是不是 CDN？”
- “你强刷了吗？”

适合搭档：LOG 日志仙

天敌人格：NaN 非数人

### 5.19 MERGE 冲突体

一句话：你走到哪里，哪里就需要手动 resolve。

典型症状：

- 每次合代码都有战争。
- 分支活得太久。
- 你和 main 永远不熟。
- 冲突文件像事故现场。

经典台词：

- “我 rebase 一下。”
- “怎么又冲突？”
- “这个是谁改的？”

适合搭档：LGTM 看过者

天敌人格：FORCE 推土机

### 5.20 REBASE 改命者

一句话：你不解决历史，你重写历史。

典型症状：

- commit history 必须漂亮。
- 不能忍 merge commit。
- 喜欢把混乱整理成从未发生。
- 同事怕你动公共分支。

经典台词：

- “我整理下提交。”
- “历史太脏了。”
- “我 rebase 一下就好。”

适合搭档：README 体面人

天敌人格：FORCE 推土机

### 5.21 FORCE 推土机

一句话：`--force` 是你的交流方式。

典型症状：

- 你不协商，你覆盖。
- 分支保护是你的人生敌人。
- 出问题后说“我以为没人用”。
- 你相信结果，不相信过程。

经典台词：

- “我 force push 了。”
- “应该没人拉这个分支吧。”
- “不行我再推一次。”

适合搭档：SUDO 强行者

天敌人格：REBASE 改命者、MERGE 冲突体

### 5.22 LOOP 死循环

一句话：你不是拖延，你是在无限迭代。

典型症状：

- 总觉得还能再改。
- 一直优化体验。
- 需求做完了，又自己加需求。
- 结束条件缺失。

经典台词：

- “我再调一下。”
- “这里还能更好。”
- “马上，真的马上。”

适合搭档：CRON 定时人

天敌人格：MVP 糊弄者

### 5.23 HEAP 堆人

一句话：什么都往里塞，迟早爆。

典型症状：

- 变量越来越多。
- 状态越来越大。
- 文件越来越长。
- 你的人生需要 GC。

经典台词：

- “先放这里。”
- “这个状态也存一下。”
- “暂时不会太大。”

适合搭档：YAK 剃牦牛者

天敌人格：TEST 测试鬼

### 5.24 STACK 栈人

一句话：压力一层一层来，最后溢出。

典型症状：

- 任务叠到爆。
- 每层问题都依赖上一层。
- 越忙越不敢停。
- 最后一个小需求把你压垮。

经典台词：

- “我排一下优先级。”
- “这个依赖前面的。”
- “先别再加了。”

适合搭档：CRON 定时人

天敌人格：PMD 产品受害者

### 5.25 ENV 环境人

一句话：在你机器上永远是好的。

典型症状：

- 本地一切正常。
- 部署一切离谱。
- 环境变量像彩票。
- 你永远差一个 `.env.example`。

经典台词：

- “我本地可以。”
- “你 node 版本多少？”
- “是不是环境变量没配？”

适合搭档：DOCKER 套娃人

天敌人格：PROD 生产祭品

### 5.26 PROD 生产祭品

一句话：只有线上环境能让你真正活着。

典型症状：

- 线上问题比咖啡更提神。
- 你知道每个报警的声音。
- 本地复现不了，线上直接看。
- 生产环境是你的真实 IDE。

经典台词：

- “线上看一下。”
- “先别慌。”
- “我开个临时日志。”

适合搭档：LOG 日志仙、HOTFIX 临终抢救者

天敌人格：BUG 虫人、ENV 环境人

### 5.27 LOG 日志仙

一句话：你不相信人类，只相信日志。

典型症状：

- 先看日志，再看人。
- 问题定位靠证据。
- 没日志的系统在你眼里是黑箱。
- 你会给关键路径埋很多 log。

经典台词：

- “日志呢？”
- “先看 trace。”
- “这个得有证据。”

适合搭档：PROD 生产祭品、CACHE 缓存怪

天敌人格：LGTM 看过者

### 5.28 TEST 测试鬼

一句话：没有测试的代码，在你眼里就是遗书。

典型症状：

- 改一行也要补测试。
- 看到手动测试会不安。
- CI 红了你会坐立难安。
- 你不是洁癖，你是经历过事故。

经典台词：

- “测试呢？”
- “这个边界覆盖了吗？”
- “先别 merge。”

适合搭档：BUG 虫人、YAML 缩进奴

天敌人格：MVP 糊弄者、ANY 任意人

### 5.29 MOCK 假人

一句话：你 mock 了世界，也 mock 了自己。

典型症状：

- 接口没好你也能开发。
- 数据全是假的，但页面很真。
- 联调时世界崩塌。
- 你活在自己构造的宇宙里。

经典台词：

- “我先 mock 一下。”
- “等后端接口。”
- “真实数据怎么长这样？”

适合搭档：CRUD 增删改查牲

天敌人格：PROD 生产祭品

### 5.30 LEGACY 遗老

一句话：你维护的不是项目，是祖坟。

典型症状：

- 代码里有十年前的注释。
- 没人敢删，你也不敢。
- 每个函数都有历史原因。
- 你知道很多不能说的故事。

经典台词：

- “这个不能动。”
- “以前踩过坑。”
- “历史原因。”

适合搭档：LOG 日志仙、TEST 测试鬼

天敌人格：YAK 剃牦牛者

### 5.31 DOCKER 套娃人

一句话：你以为你在隔离环境，其实你在隔离自己。

典型症状：

- 什么都想容器化。
- 本机不装，全靠镜像。
- 镜像越来越大。
- 调试像开盲盒。

经典台词：

- “我写个 Dockerfile。”
- “容器里跑一下。”
- “是不是 volume 没挂？”

适合搭档：ENV 环境人

天敌人格：SUDO 强行者

### 5.32 README 体面人

一句话：项目不能跑没关系，介绍一定要漂亮。

典型症状：

- README 比代码精致。
- badge 很多，功能很少。
- 截图好看，安装困难。
- 你尊重门面工程。

经典台词：

- “我先补下文档。”
- “加个 badge。”
- “这个需要一个漂亮的介绍。”

适合搭档：WIP 半成者、YAK 剃牦牛者

天敌人格：PROD 生产祭品

## 7. 隐藏人格机制

隐藏人格优先级高于普通人格。用户命中特定条件时，结果直接进入隐藏人格。

### 7.1 ENV 环境人

触发条件：

- 多次选择“我本地是好的”。
- 环境变量、Docker、部署题明显偏环境归因。

结果文案：

> 你不是有问题，你只是和世界运行在不同环境。

### 7.2 FORCE 推土机

触发条件：

- 多次选择 force push、直接覆盖、先 merge 后修。
- 风险偏好极高。

结果文案：

> 你不是在协作，你是在版本控制系统里开压路机。

### 7.3 YAK 剃牦牛者

触发条件：

- 多次选择重构、抽象、底层解决。
- 交付执念低，抽象冲动高。

结果文案：

> 你本来只是想修个按钮，现在项目已经换了构建工具。

### 7.4 404 失踪者

触发条件：

- 多次选择沉默、晚点看、装死。
- 协作状态低。

结果文案：

> 你不是不在，你只是暂时无法被 HTTP 访问。

### 7.5 AI 提示词奴

这是后续扩展人格。

触发条件：

- 多次选择“让 AI 写”。
- 多次选择复制、生成、改 prompt。

结果文案：

> 你已经不会写代码了，但你很会叫别人写。

## 9. GitHub 复判扩展

第一版可以不做 GitHub。第二版可以加入 GitHub username 分析，用公开数据做“现实复判”。

示例：

> 你自测是 TEST 测试鬼，但 GitHub 显示你的 23 个仓库里只有 1 个 test 目录。
>
> 系统复判：README 体面人。

可分析数据：

- public repo 数量
- 主要语言
- README 完整度
- 是否有 test 目录
- 是否有 CI 配置
- commit message 风格
- repo 命名风格
- 最近活跃度
- fork/star 比例
- issue/PR 参与情况

注意事项：

- 明确这是娱乐向。
- 不做能力评价。
- 不暗示不活跃 GitHub 等于技术能力弱。
- 不默认读取私有仓库。
